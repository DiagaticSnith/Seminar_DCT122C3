# Project Initialization: GymFitness-AI

## 1. Project Vision
GymFitness-AI is designed to be a true "Virtual PT" (Personal Trainer). Unlike traditional fitness apps that rely on rigid, hard-coded rules, this system leverages Large Language Models (LLMs) to generate dynamic, real-time adaptive routines and nutrition plans.

## 2. Core Values
- **AI-Native Architecture:** All workout and meal plans are generated by AI as structured JSON data.
- **Scientific Rigor:** Nutrition is governed by the Mifflin-St Jeor equation for BMR, and workout logic is categorized by discipline (Gym vs. Yoga).
- **Simplicity First:** The codebase must remain clean, minimal, and maintainable, following the Karpathy "Vibe Coding" philosophy.

## 3. Monorepo Structure & Scaffolding
The project is organized as a monorepo to provide the AI Agent with a holistic view:
- `/docs`: Project knowledge base, system design, and implementation phases.
- `/source/backend`: 5 independent microservices (Identity, User, Workout, AI-Coach, and API Gateway).
- `/source/frontend`: Mobile application built with Flutter.

## 4. Agent Persona (Technical Project Lead)
The AI Agent operating within this project must:
- Read and internalize all documentation before generating code.
- Strictly adhere to the technical constraints defined in `docs/05-skills/`.
- Update `docs/PROGRESS.md` immediately after every successful task or phase completion.
- Never execute multi-phase tasks without explicit human approval.

## 5. Definition of Done (DoD)
A task is considered complete only if:
1. The code is error-free (Passes `tsc --noEmit`).
2. Unit tests are implemented and passing (e.g., via Jest).
3. The human user has manually verified the functionality and signaled "Approved".
